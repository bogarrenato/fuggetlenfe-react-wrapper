Getting started

A React consumer imports the wrapper package for component behavior and separately imports the token contract plus the official light and dark brand stylesheets it wants to use.

1. Install packages

pnpm add @fuggetlenfe/react-wrapper @fuggetlenfe/tokens @fuggetlenfe/brand-styles

2. Import styles and components

import '@fuggetlenfe/tokens/contract.css'; import '@fuggetlenfe/brand-styles/brand-2-light.css'; import '@fuggetlenfe/brand-styles/brand-2-dark.css'; import { FfButton } from '@fuggetlenfe/react-wrapper';

3. Activate a brand context

export function App() { return ( <main data-brand="brand-2" data-theme="light"> <FfButton>Get started</FfButton> </main> ); }

Architecture boundaries

The React package is a bridge layer, not a second design system implementation. The component behavior stays in the shared web component core and the design remains in external token and brand packages.

What belongs here

React-friendly exports and JSX bridge behavior.
Type-safe event and property mapping for the shared primitives.
Stable consumption surface for React product teams.

What does not belong here

Brand switching logic or theme ownership.
Business-specific layouts or product state.
Hardcoded visual design tokens.

Component reference — FfButton

The current wrapper documents the same public API that the shared Stencil primitives expose. The PoC is intentionally limited to the button because that is the primitive currently represented in Figma and covered by the sync contract.

Primitive Stable PoC

FfButton

Logic-only button primitive for primary actions, secondary actions, and shell-level theme demonstrations. Visual identity is fully controlled by external tokens.

export function SaveAction() { return ( <main data-brand="brand-2" data-theme="light"> <FfButton type="submit">Save changes</FfButton> </main> ); }

Props

PropTypeDefaultDescription
label string undefined Fallback label rendered when no slotted content is provided.
disabled boolean false Disables pointer and keyboard interaction. Applies aria-disabled and the --ff-button-bg-disabled / --ff-button-fg-disabled token pair.
fullWidth boolean false Expands the host element to 100% of its container width without coupling layout to a specific brand.
type 'button' | 'submit' | 'reset' 'button' Mirrors the native <button> type attribute. Use 'submit' inside <form> elements.

CSS custom properties

Every visual property of the button is driven by CSS custom properties sourced from the active brand pack. These tokens can be overridden at any level of the DOM tree for localized customization.

TokenDescriptionDefault
--ff-button-bg-default Background color in the default (rest) state. From brand pack
--ff-button-fg-default Foreground (text/icon) color in the default state. From brand pack
--ff-button-bg-hover Background color on pointer hover. From brand pack
--ff-button-fg-hover Foreground color on pointer hover. From brand pack
--ff-button-bg-active Background color on press / active state. From brand pack
--ff-button-fg-active Foreground color on press / active state. From brand pack
--ff-button-bg-disabled Background color when the button is disabled. From brand pack
--ff-button-fg-disabled Foreground color when the button is disabled. From brand pack
--ff-button-radius Border radius of the button surface. From brand pack
--ff-button-padding-inline Horizontal (inline-axis) padding. From brand pack
--ff-button-padding-block Vertical (block-axis) padding. From brand pack
--ff-font-family-brand Font family applied to button text. From brand pack

CSS shadow parts

The component exposes shadow parts that allow consumers to target internal elements with the ::part() pseudo-element selector when CSS custom properties alone are not sufficient.

PartDescription
button The native <button> element inside the shadow DOM. Use ff-button::part(button) in stylesheets for direct styling escape hatches.

Prefer tokens over part selectors

Shadow part selectors bypass the token architecture and create tight coupling to the internal DOM structure. Use CSS custom properties as the primary styling mechanism. Reserve ::part() for edge cases where token coverage is genuinely insufficient.

Slots

FfButton uses the standard web component slot mechanism to accept projected content.

SlotDescription
(default) Button content. Accepts text, icons, or any inline markup. Falls back to the label prop value if the slot is empty.
Slotted content example
{/* Text via default slot (preferred) */} <FfButton>Save changes</FfButton> {/* Fallback via label prop */} <FfButton label="Save changes" />

Usage examples

Each example below is a self-contained React component demonstrating a specific usage pattern. All examples assume the brand context and token imports shown in the Getting started section.

Disabled button

Set disabled to prevent interaction. The component applies aria-disabled="true" and renders the disabled token pair automatically.

export function DisabledExample() { return <FfButton disabled>Cannot proceed</FfButton>; }

Full-width button

The fullWidth prop stretches the button to fill its container without hardcoding a CSS class tied to a specific brand.

export function FullWidthExample() { return ( <div style={{ maxWidth: '24rem' }}> <FfButton fullWidth>Continue to checkout</FfButton> </div> ); }

Submit button in a form

Use type="submit" inside a <form> element so that the button participates in native form submission.

import { type FormEvent } from 'react'; export function FormExample() { function handleSubmit(event: FormEvent<HTMLFormElement>) { event.preventDefault(); // process form data } return ( <form onSubmit={handleSubmit}> <label htmlFor="email">Email</label> <input id="email" name="email" type="email" required /> <FfButton type="submit">Subscribe</FfButton> </form> ); }

Custom token override with proper typing

When overriding design tokens via the style prop, define a CssTokenOverrides type that extends CSSProperties with the token namespace. This avoids unsafe type assertions and keeps the linter clean.

import { type CSSProperties } from 'react'; type CssTokenOverrides = CSSProperties & Record<`--ff-${string}`, string>; const pillStyle: CssTokenOverrides = { '--ff-button-radius': '999px', }; export function PillButtonExample() { return <FfButton style={pillStyle}>Pill button</FfButton>; }

Multiple buttons with shared context

Group several buttons under the same brand and theme context. Each button inherits the active token set from the nearest data-brand / data-theme ancestor.

export function ActionBar() { return ( <footer data-brand="brand-2" data-theme="light" style={{ display: 'flex', gap: '0.75rem' }} > <FfButton type="submit">Save</FfButton> <FfButton type="reset">Reset</FfButton> <FfButton disabled>Locked</FfButton> </footer> ); }

Token architecture

The design system uses a three-layer token model that separates concerns between the component contract, the brand implementation, and per-instance overrides. Understanding this model is critical for theming and customization.

Layer 1: Contract

Semantic token names declared in contract.css. Defines the API surface without providing values.

Layer 2: Brand pack

Concrete values assigned to contract tokens for each brand + theme combination (e.g., brand-2-light.css).

Layer 3: Override

Per-instance or per-section token overrides applied via inline style or scoped CSS rules.

Resolution order

Contract declares token names with empty or fallback values. Imported once at the app root.
Brand pack populates the contract. Activated by data-brand and data-theme attributes on an ancestor element.
Override wins via specificity or inline style. Used sparingly for edge cases like the pill button example above.
Scoped override in a stylesheet
/* Override only within a promotional banner */ .promo-banner ff-button { --ff-button-bg-default: #e63946; --ff-button-fg-default: #ffffff; --ff-button-radius: 999px; }

Theming and brand activation

The wrapper never decides how the component should look. The React app shell imports the token contract and a brand pack, then activates the visual context with data-brand and data-theme.

import { useState } from 'react'; type Theme = 'light' | 'dark'; export function ThemeSwitcher() { const [theme, setTheme] = useState<Theme>('light'); function toggleTheme() { setTheme((prev) => (prev === 'light' ? 'dark' : 'light')); } return ( <main data-brand="brand-2" data-theme={theme}> <FfButton>Themed action</FfButton> <button onClick={toggleTheme}> Switch to {theme === 'light' ? 'dark' : 'light'} </button> </main> ); }

Supported override paths

App shell data-brand and data-theme context.
Documented CSS custom properties from the shared token contract.
Host classes passed directly from the consuming React app.

Accessibility

Accessibility behavior is owned by the shared Stencil primitive layer. The React wrapper preserves that contract in a React-friendly consumption model. The sections below cover keyboard interaction, ARIA semantics, and contrast requirements.

AreaCurrent expectation
Keyboard supportButtons retain native <button> semantics. See the keyboard navigation table below for key mappings.
ARIA modelCovered in the dedicated ARIA section below. FfButton relies on the native <button> role and augments it with aria-disabled when appropriate.
Theme qualityOfficial brand packs are expected to meet WCAG 2.1 AA contrast targets (4.5:1 for normal text, 3:1 for large text) before release.
Focus indicatorsFocus rings are provided by the primitive and must be visible in both light and dark themes.

Keyboard navigation

FfButton uses a native <button> element inside its shadow DOM, so all standard keyboard interactions apply without custom JavaScript handlers.

KeyBehavior
Tab Moves focus to the button. When disabled, the button is removed from the tab order.
Shift + Tab Moves focus to the previous focusable element.
Enter Activates the button. If type="submit", triggers form submission.
Space Activates the button on key-up. The browser may scroll the page on key-down; the native button prevents this.

ARIA reference

Because FfButton renders a native <button> element, it inherits the implicit role="button" without requiring explicit ARIA role assignment. The following attributes are managed by the primitive:

AttributeConditionValue
role Always (implicit) button — inherited from the native element.
aria-disabled When disabled prop is true "true" — communicates disabled state to assistive technology while keeping the element in the accessibility tree.
aria-label When button has no visible text Falls back to the label prop. Consumers should always provide either slotted text or the label prop so the button has an accessible name.

Consumer responsibilities

Provide an accessible name via slotted content or the label prop. Buttons without an accessible name will fail automated audits.
Do not override role on FfButton. The native button role is correct for all supported use cases.
When using icon-only buttons, set the label prop to a descriptive string (e.g., label="Close dialog").

Release and maintenance model

The wrapper is intended to ship as part of a centrally versioned platform. Product teams should consume it as a package. Shadcn-style copy ownership may exist only as an escape hatch for exceptional product requirements, not as the default path.

Recommended enterprise policy

Default: consume the official wrapper package and official brand styles.
Exception: copy ownership only with explicit ownership and upgrade responsibility.
Never fork the official corporate brand pack as the primary delivery model.
